# max2alamo-2026 v0.9.1

**Stabilization release on top of v0.9.0.** Six writer correctness fixes informed by real-world export feedback (EI_SNOWTROOPER + RSkin meshes round-tripped through AloViewer + in-game playback), plus the test-side regen and diagnostic tooling that surfaced them. No new features; same v1.0-readiness posture as v0.9.0. Corpus round-trip remains **4,929 / 4,929** byte-identical against the vanilla EaW + FoC pool; harness runs **59 / 59 clean** against this tag.

## Quick start

1. **Download `max2alamo.dle`** from the assets below.
2. In Windows Explorer: right-click → **Properties** → **Unblock**.
3. Copy `max2alamo.dle` into a folder Max can scan — recommended: `<your-mod-project>\plugin\`.
4. In 3ds Max 2026: **Customize → Configure System Paths → 3rd Party Plug-Ins → Add**, point at that folder, restart Max.
5. Verify: **File → Export** should now list **"Alamo Object (\*.alo)"**.

Full setup details in [`docs/build.md`](https://github.com/DrKnickers/max2alamo-2026/blob/v0.9.1/docs/build.md).

## Highlights since v0.9.0

### Engine-visible writer fixes

- **Skinned meshes with RSkin / DirectX shaders render correctly.** Previously the writer emitted per-vertex `BoneIndices` as global skeleton indices and omitted the `0x10006` skin-bone-remap chunk; the engine's renderer reads `bones[bone_remap[idx]]` and saw garbage when `0x10006` was absent, producing stretched-triangle artifacts on every RSkin mesh. v0.9.1 emits `0x10006` per submesh that needs it, with `BoneIndices` switched to local-slot indices the renderer can dereference correctly.
- **Vertex format string per shader.** The `0x10002` chunk now reports the shader-appropriate vertex layout name (`alD3dVertN`, `alD3dVertNU2`, `alD3dVertRSkin`, etc.) instead of the hardcoded `alD3dVertNU2`. Renderer no longer falls back to a generic layout for shaders that need per-vertex tangents or bone weights.
- **Animation playback orientation.** Pre-v0.9.1, every emitted rotation track had its `xyz` quaternion components negated relative to the engine's convention — animations played sign-flipped. EI_SNOWTROOPER's 60 clips now visually match the authored Max scene.
- **Static accessories follow their parent bone.** A non-skinned mesh (rifle, cape, lens flare, sensor dish) that's parented in Max to a hand / chest / mount bone now inherits that bone's transform chain in the `.alo`. Pre-v0.9.1, every non-skinned mesh hardcoded to scene Root and stayed at its bind-pose world position while the rig animated around it — EI_SNOWTROOPER's rifle visibly floated free of the hand during playback.
- **Static accessories stay attached during animation.** The `.ala` static defaults (`trans_offset` + `default_rotation`) are now populated from the bind matrix for non-animatable bones — matches the vanilla `.ala` convention exactly. Pre-v0.9.1 the defaults stayed at struct zeros, so the engine read "rifle is at parent-local `(0, 0, 0)` with identity rotation" the moment any clip started playing and snapped the gun off the hand to its parent's origin.
- **Legacy `.max` clip-data import always active.** The `NOTIFY_FILE_POST_OPEN` hook that translates legacy Petroglyph appData records into Phase 11b's user-prop convention now runs unconditionally on any file open, not only when the Animation Settings rollout was open. Files opened in fresh Max sessions surface their imported clip catalogue immediately.

### Authoring guidance

- **`docs/build.md`** gained a new "Authoring convention — bind pose lives at frame 0" section. Documents two authoring traps surfaced during Phase 14 debugging:
  - `animationRange` starting after frame 0 — the walker hardcodes `igame->SetStaticFrame(0)`, so frame 0 IS the bind pose regardless of where the artist's animation actually starts. If frame 0 is outside the animation range (no keyframes there), `Position_XYZ` controllers fall back to leftover initial values and the engine renders the bind pose somewhere unexpected.
  - `BoneSys.createBone <from> <to>` with non-collinear `from`/`to` bakes a rotation into the bone's node TM derived from the from-to direction. A subsequent `b.rotation = …` keyframe then orbits the translation around the local pivot. Two safe patterns documented.

### Internal: diagnostic tooling earned its commits

The Phase 14 investigation arc produced four reusable diagnostic tools that live alongside the existing `validate_alo.py` / `alo_roundtrip` / `alo_dump` set. None affect downstream usage of the exporter; they're tools for the next person (human or Claude) investigating an export discrepancy.

- **`tools/ala_diff/`** — typed-semantic `.ala` comparator (dump-vs-vanilla side-by-side at the unpacked quaternion / translation level). Surfaces convention bugs that byte-level `alo_roundtrip` can't catch.
- **`tests/maxscript/verify/_chain_dump.py`** — given any `.alo`, walks each bone's parent chain and dumps both the encoded local TM and the chain-composed world TM. The Tier 3 oracle that lets you check "what world position does the engine reconstruct for this bone?" against Max's view.
- **`tests/maxscript/verify/_ala_dump_bone.py`** — single-bone `.ala` track dumper. Shows track metadata (`idx_translation`, `idx_rotation`, `trans_offset`, `trans_scale`, `default_rotation`) + per-frame reconstruction. Surfaces "engine reads this constant value because of the `trans_offset` field, not because of the bind matrix" subtleties.
- **`tests/maxscript/_phase14d_*.ms`** — synthetic single-bone export variants + EI_SNOWTROOPER chain-dump + local-invariance check scripts. Reusable pattern for any user-content investigation.

### Testing

- Corpus round-trip: **4,929 / 4,929 PASS** (1,383 + 683 `.alo` + 1,500 + 1,363 `.ala`) — exact match to v0.9.0 baseline. The writer fixes are in `scene_walker.cpp` (Max-side), not in the format-library reader/writer, so vanilla byte-identity is unaffected.
- Max-side harness: **59 / 59 PASS** against the v0.9.1 tag (modulo the unrelated `test_phase14_min_chain` missing-verifier note that pre-dates this release).
- Catch2 unit tests: unchanged from v0.9.0 (159 cases / 927 assertions).

## Known limitations

Carried forward from v0.9.0:

- The `.dle` is built locally and attached manually per release — the Max 2026 SDK is non-redistributable, so CI cannot ship the plugin. See [`docs/build.md`](https://github.com/DrKnickers/max2alamo-2026/blob/v0.9.1/docs/build.md) to build from source.
- Per-mesh `_ALAMO_VERTEX_TYPE` user-prop override remains unwired. v0.9.1's Phase 10 fix unblocked per-shader vertex-format-string emission; the per-mesh override path is still tracked as a post-v0.9 item ([#75](https://github.com/DrKnickers/max2alamo-2026/issues/75)).
- AloViewer / Mike Lankamp's importer round-trip remains a manual Tier 4 smoke (no headless AloViewer).

## Reporting issues

Use the GitHub issue tracker. Particularly useful in an issue report:

- Your Max version (must be 2026 for the `.dle`).
- The `.export.log` sidecar from a failing export.
- For round-trip failures: the original `.alo`, plus `alo_dump` output if you can produce it.
- For animation playback issues: a `.ala` from a single representative clip + a description of what you see in AloViewer vs Max GUI.
